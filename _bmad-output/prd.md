# Product Requirements Document - leon-blog

**Author:** Leon  
**Date:** 2025-12-22

---

## Executive Summary

### 项目概述

**leon-blog** 是一个面向独立软件开发专家、AI4SE 软件工程转型专家及阅读爱好者的现代化个人技术博客平台。基于 Next.js 构建，采用类似 benx.ai 的极简专业设计风格，目标是建立个人品牌影响力，最终成为连接外部资源和商业化的入口。

### 目标用户

| 用户群体 | 核心需求 | 价值主张 |
|---------|---------|---------|
| **IT从业者** | 学习AI编程工具、提升开发效率 | 实战案例、可复制的方法论 |
| **技术团队管理者** | AI工具落地方案、团队效能提升 | 已验证的实践经验、咨询服务 |
| **企业管理者** | 数字化转型路径、AI应用场景 | 战略视角、成功案例背书 |

### 核心定位

> **"用AI，做有意思的事"** — 全栈开发者 × Vibe Coding 实践者 × 超级个体探索者

从产品经理转全栈开发者，见证移动互联网到AI时代的变迁，致力于帮助更多人拥抱AI时代的变化，从执行者进化为构建者。

### 商业愿景

**短期目标 (0-6个月)**：
- 建立专业形象，验证内容市场需求
- 积累种子用户和订阅者
- 通过轻量咨询验证付费意愿

**中期目标 (6-12个月)**：
- 形成稳定的内容输出节奏
- 咨询服务规模化
- 课程/培训产品开发

**长期愿景 (12个月+)**：
- 成为AI编程领域的意见领袖
- 建立付费社群和会员体系
- 多元化商业变现

### MVP 范围定义

#### P0 功能（必须实现）

| 模块 | 功能 | 说明 |
|------|------|------|
| **首页** | Hero区域 + 个人介绍 | 中英文客户端切换（仅首页）|
| **首页** | 关于我 | 详细背景、职业轨迹、生活爱好 |
| **首页** | 专业服务 | 产品设计咨询、AI编程课程 |
| **首页** | 项目经历 | WaytoAGI社区、飞书多维表格等 |
| **首页** | 教程分享 | 案例分享入口 |
| **博客** | 文章列表 | 卡片式布局、分类标签 |
| **博客** | 文章详情 | MDX渲染、代码高亮 |
| **博客** | 搜索功能 | 标题/内容搜索 |
| **博客** | RSS订阅 | 标准RSS Feed |
| **全局** | 暗色模式 | 亮/暗主题切换 |
| **全局** | 响应式设计 | 移动端优先 |

#### P1 功能（建议实现）

- 邮件订阅功能
- 阅读进度指示器
- 目录导航（TOC）
- 代码复制按钮
- 文章阅读时间估算

#### P2 功能（未来迭代）

- 评论系统
- 阅读统计
- 付费内容墙
- 会员系统
- 后台管理系统

#### 明确不做

- 博客中英文切换（仅首页需要）
- 复杂的i18n路由
- 后台管理系统（MVP阶段使用Git管理内容）
- 用户注册/登录系统

### 技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| **框架** | Next.js (App Router) | SSG/SSR混合、SEO友好、生态成熟 |
| **样式** | Tailwind CSS | 快速开发、一致性、暗色模式 |
| **内容** | MDX + Contentlayer | Markdown + React组件 |
| **部署** | Vercel | Next.js原生支持、边缘网络 |
| **i18n** | 客户端切换 | 仅首页、无复杂路由 |
| **暗色模式** | next-themes | 持久化用户偏好 |

### 差异化优势

1. **真实案例驱动**：不是理论分享，而是实战经验的沉淀
2. **从执行者到构建者**：独特的职业转型叙事
3. **中文社区聚焦**：服务中文技术社区，差异化于英文博客
4. **超级个体视角**：AI时代个人能力边界的探索

### 关键风险与缓解

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| 内容同质化 | 用户流失 | 聚焦实战案例、透明化输出 |
| 更新断档 | 品牌受损 | 提前储备内容、建立创作节奏 |
| 商业化过早 | 用户反感 | 先建立信任，后自然转化 |
| 技术过度设计 | 延迟上线 | MVP优先、迭代改进 |

### 成功画像

访客说"这正是我需要的"的时刻：
- 看到真实的项目案例和可验证的成果
- 第一眼感受到简洁专业且亲和的设计
- 找到可以直接复用的AI编程方法

用户意识到"不一样"的时刻：
- 简单整洁，以人为本，态度认真
- 不是空洞的理论，而是实践者的分享

产品大获成功时用户的改变：
- 真正学会使用AI提高工作效率
- 从繁杂的执行工作中抽身
- 从执行者进化为构建者

### 指标分层关注策略

**🟢 日常追踪（每周）— 可控层**
- 内容发布：目标周更
- 连续更新周数：保持不断更
- 内容储备：≥2周存稿
- 时间投入：≤15小时/周

**🟡 月度复盘（每月）— 增长层**
- 订阅者增长趋势
- 咨询表单/私信数量
- 月访问量
- 内容转化率：每篇内容带来的咨询数

**🔵 季度验证（每季度）— 结果层**
- 付费客户数
- 客户案例数
- 副业收入占比
- 客户终身价值

---

### 诊断性指标

| 指标 | 计算方式 | 用途 |
|------|---------|------|
| 获客效率 | 客户数 ÷ 内容生产时间 | 判断内容ROI |
| 咨询转化率 | 咨询数 ÷ 访问量 | 定位流量问题 |
| 付费转化率 | 付费数 ÷ 咨询数 | 定位转化问题 |
| 时间杠杆率 | 长期收益 ÷ 每小时投入 | 判断效率 |

---

### 可持续性保障

| 维度 | 标准 | 预防风险 |
|------|------|---------|
| 内容储备 | 始终≥2周存稿 | 断更风险 |
| 更新节奏 | 周更而非日更 | 精力耗尽 |
| 时间边界 | 每周≤15小时 | 生活平衡 |
| 品牌清晰度 | 一句话定位被70%读者认可 | 定位模糊 |

## User Journeys

### Journey 1: 张磊 - 技术团队管理者的AI转型困局

**角色背景：**
> 张磊，35岁，某互联网公司技术总监，管理20人研发团队。公司刚拿到新一轮融资，CEO要求半年内实现"AI驱动的研发效能提升50%"。张磊压力山大——他知道AI编程工具很火，但不知道如何落地，更担心贸然引入会打乱团队节奏。

**📍 Opening Scene: 深夜的焦虑**

周五晚上11点，张磊还在公司。刚结束的周会上，CEO再次追问AI落地进展。他打开浏览器，搜索"团队如何引入AI编程工具"，满屏都是Cursor、Copilot的个人使用教程，但没有一篇告诉他：作为管理者，应该关注什么指标？怎么评估效果？怎么防止代码质量下滑？

他在小红书刷到一篇笔记：《AI时代，技术管理者应该关注什么？》，作者是一个叫Leon的人。他点进了 leon-blog。

**📍 Rising Action: 发现价值**

首页的一句话抓住了他："从产品经理转全栈，边做边学，努力成为超级个体"

他点开博客，看到几篇文章让他边看边截图发给CTO。文章最后有一句话："如果你正在推动团队的AI转型，欢迎聊聊，我踩过的坑可能帮你省几个月。"

**📍 Climax: 决定联系**

张磊点开"专业服务"页面，加了微信，发了消息请求咨询。

**📍 Resolution: 新的开始**

三天后的视频咨询中，Leon帮张磊梳理了"3阶段试点方案"。六个月后，张磊的团队成为公司AI转型的标杆。

**旅程1触点与功能需求：**
- 社交媒体引流 → 博客内容便于分享
- 首页第一印象 → 清晰的个人定位
- 博客浏览 → 分类、搜索、阅读体验
- 转化引导 → 专业服务页面、联系方式

### Journey 2: 陈总 - 企业管理者的决策困境

**角色背景：**
> 陈明远，45岁，传统制造业企业副总裁，分管IT和数字化。公司年营收5亿，IT团队只有8人，外包依赖严重。董事会要求评估"AI能否帮公司降低软件开发成本"。

**📍 Opening Scene: 董事会的压力**

月初的董事会上，投资人问："现在AI编程这么火，你们的软件开发成本能降多少？"陈总说不出具体数字。

**📍 Rising Action: 偶然发现**

陈总在企业家微信群里看到一篇文章链接：《AI时代，企业应该怎么评估软件开发成本？》。文章来自leon-blog，给出了评估框架，让他眼前一亮。

**📍 Climax: 寻求专业支持**

IT负责人建议约一次企业咨询。陈总同意了。

**📍 Resolution: 董事会的转变**

咨询后，Leon帮陈总团队做了《AI对企业软件开发成本影响评估报告》。下个月的董事会上，陈总展示了评估报告，给出了具体预测。一年后，公司的外包成本降了60万。

**旅程2触点与功能需求：**
- 口碑传播 → 内容便于分享
- 快速建立信任 → 项目经历、成功案例

- 高端转化 → 支持团队代为联系

## Innovation & Novel Patterns

### 核心叙事：全栈开发者 → 技术商人

**传统路径**：技术专家 → 更资深的技术专家  
**leon-blog路径**：全栈开发者 → 技术商人（Tech Businessman）

> 不是"写代码赚钱的人"，而是"用技术能力解决商业问题、直接创造和捕获价值的人"

---

### Detected Innovation Areas

#### 1. Build in Public 完全透明化

| 维度 | 传统做法 | leon-blog创新 |
|------|---------|--------------|
| 成功案例 | 精选展示 | 全部公开 |
| 失败经历 | 隐藏 | 透明分享 |
| 收入数据 | 保密 | 公开披露 |
| 趋势预测 | 无复盘 | Track Record验证 |

**风险应对**：设定透明边界、框架化失败、用成果压过噪音

#### 2. 跨界叙事融合

将四个领域融合为统一视角：产品思维 × 技术实践 × 团队管理 × 企业经营

**对外翻译（场景化定位）**：
- 对IT从业者："帮你学会AI编程，拓宽能力边界"
- 对技术管理者："帮你落地AI工具，解决团队问题"
- 对企业管理者："帮你评估AI影响，支持决策"

#### 3. 内容分层策略

**流量层（IT从业者）**：实用教程、转型故事、低价产品入口

**转化层（技术管理者）**：落地方案、踩坑清单、15分钟免费初聊

**高价值层（企业管理者）**：一页纸摘要、背景背书、ROI明确的咨询

#### 4. 跨界模式嫁接

| 来源 | 模式 | 应用 |
|------|------|------|
| 独立游戏 | Early Access | 咨询抢先体验、共创迭代 |
| 健身教练 | 陪跑模式 | 90天落地计划、定期复盘 |
| 投资通讯 | Track Record | 趋势预测复盘、验证判断力 |
| 脱口秀 | 个人声音 | 有态度观点、叙事驱动 |

### Strategic Roadmap

#### 2025验证年 · 6个月关键动作

| 里程碑 | 目标 |
|-------|------|
| 博客上线 | 完成MVP功能 |
| 内容验证 | 发布10篇高质量文章 |
| 意向积累 | 获得10+个咨询意向 |
| 商业验证 | 完成1个付费咨询 |
| 转化路径 | 形成内容-咨询飞轮 |

#### 核心原则

- **项目优先，内容为辅** — 不做"内容生产机器"
- **案例是硬通货** — 每个咨询产出一个可公开案例
- **"见过"比"会做"值钱** — 经验的价值在于踩过的坑
- **超级个体是阶段** — 让成果说话，而非标签宣称

---

### Innovation Risk Mitigation

| 风险 | 应对策略 |
|------|---------|
| 透明化负面反应 | 设定边界、框架化失败、不争论 |
| 6个月无付费 | 分层诊断、先积累案例再收费 |
| 定位不被理解 | 场景化翻译、问题导向表达 |
| 内容快速过时 | 多写"为什么"，少写"怎么做" |

| Track Record | 预测复盘 | 判断准确率 |

### Technical Architecture

#### 渲染策略

| 页面类型 | 渲染方式 | 说明 |

#### 技术选型决策

| 决策点 | 选择 | 备注 |
